Using Mealy Machines to model responsive enterprises of early 21st century

Responsive Enterprises

Anyone who knows my code knows how much I reach for modeling using finite state machines aka FSMs (often Mealy Machines; yep the state monad among other things) in many contexts in software, but in this case it is an even more scintillating application because authors Meijer and Kapoor create the analog of responsive enterprises to a closed-loop feedback system, which is basically a Mealy Machine. This is simultaneously obvious yet genius...

This is a thoughtful exploration of how every enterprise will need to become a software business to survive in the next decade and remain competitive in their field. It compares and contrasts how traditional enterprises manage process and people to software businesses.

Even within software businesses I have worked in during the course of my career I have seen executive management miss the mark on managing both aspects (iterative feedback process and the engineering culture) but especially the people risks (the "Hacker Way") outlined in this ACM article.

The Responsive Enterprise: Embracing the Hacker Way

Codifying Business Process

For the code inclined here is some Scala code to represent a Mealy machine:

type Mealy[S, I, O]= (I, S) => (O, S)

As you can see we have a parameterized type that depends on the current accumulated state value of type S, which based on some logic that needs to be defined by those that model the state machine and the current input of type I, to yield a new state value of same type S for the next round plus the output of type O.

It's amazing how many software engineers use overly complicated object oriented (OO) design patterns instead of the most appropriate abstraction to model their domains when [finite] state machines are most suitable, but I will leave that there before I diverge into a totally separate topic. (But suffice it so say functional programming does offer a more natural way to define FSMs over OO's clunky representations).

In this context we receive input from external stimuli/metrics/measurements to feed into the closed loop system along with the current business "memory" (state) which is accumulated in some hopefully meaningful way to produce a new state value and output.

This makes the business process iteratively reactive. Meijer and Kapoor point out at times state/memory resets are necessary to avoid catastrophic business failure and that would be represented in the mealy machine definition itself of the business. Genetic algorithms have been investigated in recent research with inferring finite state machines (FSMs) which is no coincidence...moving on to the more problematic concern.

Human Factors

The ACM article above really highlights some great gotchas for businesses evolving into software companies to avoid common pitfalls. It's a must read for any manager: executive or middle. And entrepreneur.

If Erik Meijer has written an article or paper where he misses the mark, I have yet to read it.
